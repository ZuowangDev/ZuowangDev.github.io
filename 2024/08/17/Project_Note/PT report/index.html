Global Timing Summary Report

  • command: report_global_timing: Top Level Summary of the timing report
    • Worst negative slack(WNS) among all endpoints
    • Total negative slack(TNS) of all endpoints
    • Number of violating endpoints

Path Timing Report

report timing provides detailed timing information about any number of requested paths.

  • Gate levels in the logic path

  • Incremental delay value of each gate level

  • Sum of the total path delays

  • Amount of slack in the path

  • Source and destination clock name, latency, and uncertainty

  • On-chip variation (OCV) with clock reconvergence pessimism removed

report_timing

  1. -significant_digits: 显示的
  2. 当-max_paths设置为大于1的任何值时，命令仅报告具有负松弛的路径。要在多路径报告中包含正松弛路径，请使用-slack_lesser_than选项。
  3. 可以在单个命令中使用多个-through选项来指定穿越多个点的路径：report_timing -from A1 -through B1 -through C1 \ -to D1
  4. report_timing -transition_time -capacitance:report transition time and capacitance

Report Normalized Slack

路径的最大频率取决于路径的传播延迟和路径所需的时钟周期数。要确定现有设计的可实现频率，您需要找到限制频率的计时路径，并专注于优化这些路径。PrimeTime通过计算归一化松弛（计时路径的度量）来找到对时钟频率影响最大的路径。PrimeTime使用以下方程计算路径的归一化松弛：
$[Normalized slack] = [Path slack] / [Allowed propagation delay for path]$

要启用规范化松弛分析，请在运行定时分析之前将timing_enable_normalized_slack变量设置为true。启用规范化的松弛分析后，report_timing命令会报告规范化的松弛数据。该工具还可以按归一化松弛对路径进行排序，并找到归一化松弛最差的路径。要报告最糟糕的标准化松弛，请输入report_timing -normalized_slack

Setting Limits for Normalized Slack Analysis

可以通过将沿路径允许的传播延迟限制在时钟周期的指定倍数来减少效果。要指定此倍数，设置timing_max_normalization_cycles变量, The default is four clock cycles.

-exclude 排除路径: 将pin、port、cell等指定part排除，优先级高于-from -to

Quality of Result Report

  • Length of the worst paths (shown as “Critical Path Length” in the report, which represents the arrival time of the signal at the path’s endpoint)

  • Total negative slack (TNS)

  • Minimum delay and hold

  • Detailed design rule constraints summary

Total negative slack (TNS) kinds

  • Union TNS mode (default) – The tool adds up the worst slack for all endpoints with negative slack, taking the worst-slack values across all path groups, across all scenarios. Each endpoint with negative slack contributes only once to the TNS, even if that endpoint has negative slack in multiple path groups (for example, when two clocks reach the same endpoint).

  • Every-group TNS mode – The tool adds up the worst slack for all endpoints with negative slack per path group, across all scenarios. If a given endpoint exists in two path groups and the endpoint has negative slack in both groups, it contributes twice to the TNS.

You select the mode by setting the timing_report_union_tns variable to true for the union mode (the default) or false for the every-group mode.

Constraint Reports

report_constraintreport the constraint violations
PT can report the maximum area of a design and certain timing constraints

The followings are some kinds of constraints:

Timing Constraints

  • Maximum path delay and setup

  • Minimum path delay and hold

  • Recovery time, the minimum amount of time required between an asynchronous control signal (such as the asynchronous clear input of a flip-flop) going inactive and a subsequent active clock edge

  • Removal time, the minimum amount of time required between a clock edge that occurs while an asynchronous input is active and the subsequent removal of the asserted asynchronous control signal

  • Clock-gating setup and hold

  • Minimum pulse width high or low at one or several clock pins in the network

  • Minimum period at a clock pin

  • Maximum skew between two clock pins of a cell

Design Rule Constraints

  1. to ensure that the netlist meets the design rules defined by the ASIC vendor and specified in the design library.
  2. can also set the important parameter such as set_max_capacitance.

  • Total net capacitance (minimum and maximum limits)

  • Pin transition time (minimum and maximum limits)

  • Sum of fanout load attributes on net (minimum and maximum limits)
    That are default generation, can also set the design rule constraint separately for rising and falling transitions, capacitance, and different clock domains and data paths for these clock domains.

To show detailed information about the violations, use the report_constraint -verbose

Maximum Skew Checks

there are two key words to express the type og Skew checks :

  • skew_rising
  • skew_falling
    this key word can be set in Library Compiler
  • skew_rising: intrinsic_rise: maximum skew time between ref pin rising to the constrained pin rising; intrinsic_fall: maximum skew time between the reference pin rising to the constrained pin falling.
  • skew_falling: intrinsic_rise: the maximum skew time between the reference pin falling to the constrained pin rising; intrinsic_fall: the maximum skew time between the reference pin falling to the constrained pin falling.
    PT: report_constraint -max_skew
    report all the violators of max skew: report_constraint -max_skew -all_violators

No-Change Timing Check

Sertain signals need no-change timing checks to make sure that they do not switch during the active interval of a periodic signal such as a clock

Mininum Pulse Width Report

report_min_pulse_width: set_timing_derate -late 2.0 -cell_check
report_min_pulse_width -all_violators -path_type full_clock -derate

Minimum Period Report

report_min_period
If there is a minimum period violation, you can debug the problem by reporting the full clock path:
report_min_period -path_type full_clock_expanded

Bottleneck Report

  • report_bottleneck
      1. If you use the report_bottleneck command in your flow, set the timing_save_pin_arrival_and_slack variable to true before your first timing update.

Global Slack Report

report_global_slack:Before using this command, time the design with the timing_save_pin_arrival_and_slack variable set to true.

Analysis Coverage Report

report_analysis_coverage: Generating information about timing checks is most critical for new designs
Analysis coverage checks are critical for sign-off. Follow this basic flow:

  1. Runlink_design and resolve link errors.
  2. Runcheck_timing and resolve timing check errors.
  3. Run report_analysis_coverage and resolve untested issues.
  4. Perform the rest of the analysis.

report_analysis_coverage:

  • Setup

  • Hold

  • No-change

  • Minimum period

  • Recovery

  • Removal

  • Minimum pulse width

  • Clock separation (master-slave)

  • Clock-gating setup

  • Clock-gating hold

  • Output setup

  • Output hold

  • Maximum skew

Clock Network Timing Report

timing characteristics of the clock network: report_clock_timing

Latency and Transition time Reporting

The clock latency at a particular pin depends on the followings analysis conditions:

  • Constraint type: setup or hold

  • Timing path role: launch or capture

  • Transition type: rise or fall

Skew Reporting

  • single clock: report_clock_timing -type skew
  • different clocks: report_clock_timing -type interclock_skew

PT considers the following conditions at each pin:

  • Type of sequential device involved: rising or falling edge-sensitive; or high or low level-sensitive

  • Type of constraint: setup or hold (which determines the path type, transition type, and library)

  • Role of the sequential device in the constraint relationship: launch or capture

Interclock Skew Reporting

To report skew between pins clocked by different clocks as well as by the same clock, use the report_clock_timing -type interclock_skew

  • Skew between pin A (clocked by CLK1) and pin B (clocked by CLK2)

  • Worst local skew between all sequential devices clocked by CLK1 and all sequential devices clocked by CLK2

  • Ten worst skews to all devices that communicate with pin A, irrespective of their domain

Clock Timing Reporting Options

report_clock_timing -type:

  • summary: minimum and maximum values for each clock
  • skew -nworst 5: Skew report, pin pairs and associated skew, one clock per pair
  • interclock_skew –nworst 5: Interclock skew report (pin pairs and associated skew, any clocks)
  • transition –nworst 5: Pin report (single pins and associated latency, transition time)
  • report_clock_timing -from U1/CP -to U5/CP -verbose -type skew: Verbose report (detailed timing along clock paths)

Summary Rreport

A summary report shows a list of the minimum and maximum latency, transition time, and clock skew values found in the requested scope of the clock network.

  • Pins at which the minimum or maximum value occurred

  • Minimum or maximum time value

  • Conditions under which the minimum or maximum time value occurred

List Report

A list report provides latency, transition time, and skew information in greater detail than a summary report.

  • skew report
  • pin report

eg: report_clock_timing -type interclock_skew \ –nworst 12 -setup -include_uncertainty_in_skew
The report shows the 12 largest skew values between clock pins, whether clocked by the same clock or by different clocks.

Verbose Path-Based Report

The most detailed type of clock network timing report is the verbose, path-based report. This type of report shows the calculation of skew, latency, and transition times along the clock path.
report_clock_timing \ -from f3_3/CP -to f2_2/CP -verbose -hold -type skew -include_uncertainty_in_skew
This command produces a report showing the transition time, incremental delay, and total latency along the clock paths to the specified pins; and the clock skew between the two pins
